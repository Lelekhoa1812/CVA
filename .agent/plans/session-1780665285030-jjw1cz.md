## Intent, Scope, and Non-Goals
Intent:
- Add a responsive `/auto-apply` page and supporting backend that lets authenticated users search for jobs, rank fit against selected ground truth, upload a resume/supporting document, prepare application materials, answer employer questions conservatively, and review before final confirmation.
- Reuse the existing search API route (`src/app/api/search/jobs/stream/route.ts`) and cover-letter generation route (`src/app/api/generate/cover-letter/route.ts`) instead of rebuilding discovery or cover-letter generation.
- Align AI usage with the repository’s current Azure/OpenAI wrapper (`src/lib/ai.ts`) while updating preset selection so the stronger reasoning path can use `gpt-5.4` and lightweight extraction/classification can use `gpt-5.4-mini`.

Scope:
- New route/page: `/auto-apply` plus navbar entry and mobile short label.
- New persistence for session/job/application/memory state in MongoDB via Mongoose, matching existing model conventions (`src/lib/models/*.ts`).
- API routes under `/api/auto-apply/*` for session creation, retrieval, update, search attach, ranking, uploads, grounded answer generation, review package creation, explicit submit gating, and event retrieval.
- A server-side orchestration layer with typed service functions for supervisor decisions, ranking, dedupe, answer grounding, compliance fallback, and activity logging.
- Resume/supporting-document ingestion reusing current multipart/file parsing patterns from `src/app/api/ocr/route.ts` and `src/lib/profile-import.ts`.
- Tests covering ranking/dedupe, answer grounding, clarification triggers, and submission confirmation gates alongside the current lightweight node test harness in `tests/career/run-tests.ts`.

Non-goals:
- Do not implement stealth automation, captcha solving, login bypassing, anti-bot evasion, paywall circumvention, fake account creation, or automatic submission without explicit confirmation.
- Do not add Playwright/Puppeteer/Selenium or any new browser-automation dependency, because no such runtime/tooling exists in `package.json` and the repository has no current browser-agent implementation surface.
- Do not rebuild the search crawler or duplicate cover-letter prompt logic already owned by the existing search and cover-letter routes.
- Do not introduce a broad long-term memory system outside the bounded `AutoApplyMemory` model; reusable profile memory should remain opt-in and session-first.
- Do not mutate existing control-room/search persistence semantics beyond shared utility reuse where needed.

Assumptions:
- “Browser preview” can ship first as a transparent application-preview/activity panel backed by extracted job/application state and placeholder/manual-guided previews, because the repository currently lacks browser automation infrastructure.
- File uploads can be stored in Mongo documents or GridFS-like metadata later, but the smallest coherent first implementation is a Mongo model storing file metadata plus extracted text/base64 or a capped content payload only as needed for current flows.
- The initial feature should prefer manual guided fallback for restricted sources, consistent with `blockedSources` already emitted by the search stream route.

## Detailed Solution Design
### Frontend UI design and construction
- Add `cv-assistant/src/app/auto-apply/page.tsx` as a client page composed from existing `ModuleShell`, `GlassPanel`, `SectionHeading`, and current gradient/glass classes so the feature visually matches Search, Generate, and Control Room.
- Structure the page as a left-to-right responsive workspace rather than a wizard-only form:
  - Top hero/status band: session title, current state chip, save status, last action, explicit “Human-in-the-loop” banner.
  - Left rail / mobile accordion: session setup, profile completeness, selected evidence, uploaded files, browser-control status, clarification queue.
  - Main center column: searchable job list + selected job detail + generated preparation/review panels.
  - Right rail / bottom-on-mobile: timeline/event feed, memory summary, policy notices, and browser preview checklist.
- Reuse the Search page’s streaming patterns for the discovery block:
  - same `SearchRequest`-shaped form state,
  - same source progress rendering,
  - same result cards/dedupe handling,
  - same abort/reset behavior,
  - but add “Attach to session” and “Prepare application” actions per result.
- Reuse Generate page patterns for evidence controls:
  - load profile once,
  - show ranked evidence cards,
  - allow manual project/experience toggles,
  - show cover-letter generation status/result,
  - keep export/review actions co-located with generated artifacts.
- Borrow Control Room patterns for inspection surfaces:
  - timeline of `ApplicationEvent`-like steps,
  - detail cards with scores/gaps/warnings,
  - review drawer/panel for artifacts and rationale,
  - compact recommendation chips and severity classes.
- Mobile behavior:
  - collapse rails into stacked sections,
  - pin the primary action bar to bottom on narrow screens,
  - keep search progress and review-confirmation actions visible without horizontal overflow,
  - avoid hover-only affordances.
- Major UI states to explicitly design:
  - empty session,
  - searching,
  - search complete with zero results,
  - result attached and being analyzed,
  - waiting for clarification,
  - review ready,
  - browser/manual-submit in progress,
  - submitted / paused / blocked.
- Browser control UI should not pretend hidden automation exists. Instead the page should visibly show:
  - current target URL,
  - form fields the agent believes need values,
  - proposed answers and evidence citations,
  - controls like “Open listing”, “Mark step complete”, “Answer manually”, “Ask agent to reconsider”, and “Confirm submit”.
- Add explicit, repository-convention technical comments near major new orchestration/UI sections using `Motivation vs Logic` because the user requested them for major feature changes.

### API design
- Add a bounded `/api/auto-apply/*` namespace using Node runtime routes and existing auth/db helpers.
- Proposed routes and responsibilities:
  - `POST /api/auto-apply/sessions`: create a new session from selected search/profile defaults.
  - `GET /api/auto-apply/sessions/:id`: fetch the fully serialized session, attached jobs, artifacts, clarifications, memory summary, and event history for page hydration/reload.
  - `PATCH /api/auto-apply/sessions/:id`: update session preferences, policy toggles, selected evidence, manual answers, or state transitions that are user-authored.
  - `POST /api/auto-apply/sessions/:id/search`: submit a `SearchRequest`-compatible payload and either proxy the existing `/api/search/jobs/stream` contract or server-side reuse the same search services while attaching results to the session.
  - `POST /api/auto-apply/sessions/:id/jobs`: attach/select one streamed search result into the session domain for preparation.
  - `POST /api/auto-apply/sessions/:id/rank`: run local heuristic + optional model reasoning to rank session jobs against profile/context evidence.
  - `POST /api/auto-apply/sessions/:id/uploads`: reuse OCR/profile-import parsing for resume/supporting documents and persist extracted text plus metadata.
  - `POST /api/auto-apply/sessions/:id/prepare`: generate preparation package, requirement map, ATS warnings, and recommended answers.
  - `POST /api/auto-apply/sessions/:id/clarify`: record user answers to clarification questions and resume the workflow.
  - `POST /api/auto-apply/sessions/:id/review`: assemble the final review package, including cover letter, evidence, warnings, browser steps, and confirmation requirements.
  - `POST /api/auto-apply/sessions/:id/browser-step`: persist manual browser observations/actions such as current URL, extracted fields, completed tasks, or blocked states.
  - `POST /api/auto-apply/sessions/:id/confirm-submit`: require explicit user confirmation payload before persisting a submitted state/event.
  - `GET /api/auto-apply/sessions/:id/events`: paginated/reload-safe event feed if the page needs separate timeline polling.
- Every route should follow existing repo conventions:
  - authenticate with `getAuthFromCookies`,
  - connect with `connectToDatabase`,
  - return JSON only,
  - serialize Mongoose docs into plain objects,
  - prefer small route files delegating into lib services.

### Service layer, tool calls, and agent reasoning
- Create a dedicated service/orchestration surface under `cv-assistant/src/lib/auto-apply/` rather than embedding workflow logic inside routes.
- Prefer a bounded supervisor pattern similar to `career/workflow.ts`:
  - session loader,
  - ranking/preparation service,
  - clarification detector,
  - browser-step interpreter,
  - review-package builder,
  - submit gatekeeper.
- Tool-call model should be explicit and local-first, not speculative “autonomous browser magic”. The agent’s callable tools/services should be limited to repository-backed capabilities such as:
  - `searchJobs` → existing streamed search route/service,
  - `loadUserProfile` / `loadUserContextSnapshot`,
  - `parseUploadedDocument` → existing OCR/profile-import helpers,
  - `rankJobFit` → reuse ATS/coverage/liveness helpers and optional AI summarization,
  - `generateCoverLetter` → existing cover-letter route/helper contract,
  - `buildReviewPackage` → local assembler,
  - `recordEvent` / `persistMemory` / `updateBrowserState`.
- Reasoning policy:
  - first collect deterministic evidence from profile, selected job, extracted requirements, uploaded docs, and prior session memory,
  - use heuristics/helpers for requirement matching and ATS checks,
  - call AI only for bounded synthesis/extraction where deterministic logic is insufficient,
  - persist the resulting rationale, confidence, and unresolved gaps so the UI can show why something happened.
- Expected reasoning outputs per stage:
  - Search attach: normalized job facts, liveness status, dedupe key, blocked/manual-source note.
  - Ranking: score, requirement coverage, supporting evidence references, missing evidence, confidence.
  - Clarification: exact missing fact/question, why it is needed, whether the workflow can continue conservatively without it.
  - Preparation: draft answers, evidence citations, ATS risks, unsupported-claim warnings.
  - Review: final package summary, policy gates, required human confirmations.
- Tool selection should prefer already-local helpers before any model call:
  - requirement matching from `career-strategist-helpers.ts`,
  - ATS validation from `ats.ts`,
  - liveness from `liveness.ts`,
  - cover-letter evidence normalization from `cover-letter.ts`.

### Memory management design
- Keep memory bounded, inspectable, and session-first.
- Introduce a dedicated session memory model only for data that does not fit existing lead/artifact/event models, e.g. `AutoApplySession`, `AutoApplyTask`, `AutoApplyMemory`, `AutoApplyBrowserState`.
- Suggested memory buckets:
  - immutable snapshot memory: selected profile facts/evidence at session creation time,
  - working memory: ranked jobs, extracted requirements, pending questions, current browser form fields, draft answers,
  - decision memory: why the agent ranked/asked/blocked/continued,
  - confirmation memory: explicit user approvals, final submission acknowledgement,
  - replay memory: timeline/event rows for reconstruction and debugging.
- Memory policy:
  - store concise summaries and references, not unbounded chain-of-thought,
  - cap raw page/form snapshots and uploaded-text payload sizes,
  - retain evidence citations linking back to profile items or uploaded documents,
  - mark machine-inferred values vs user-confirmed values separately,
  - allow session reset/reopen without mutating unrelated career/control-room data.
- Existing persistence to reuse wherever possible:
  - `ApplicationEventModel` for timeline-style events when semantics overlap,
  - `GeneratedArtifactModel` for cover letters/review exports if the artifact shape fits,
  - `JobLead` relationships for attached search results.

### Task policies and clarification behavior
- Policies must be first-class persisted config, not just prompt text. Each session should track booleans/limits such as:
  - requireHumanConfirmationBeforeSubmit = true,
  - allowGeneratedAnswersForEmployerQuestions = true/false,
  - allowUseOfUploadedResumeFacts = true,
  - disallowUnsupportedClaims = true,
  - stopOnMissingMandatoryField = true,
  - maxAutoAnsweredQuestionsBeforePause,
  - restrictedSources/manualOnlySources.
- Clarification behavior should be deterministic and user-visible:
  - ask for clarification when the job asks for missing mandatory facts, when evidence confidence is low, when uploaded/profile facts conflict, or when a generated answer would require invention.
  - each question should include `question`, `reason`, `blocking`, `suggestedAnswer?`, and `acceptableFallback?`.
  - non-blocking gaps may proceed with conservative placeholders or manual-review warnings; blocking gaps should pause submission preparation.
- Expected behaviors:
  - If profile evidence is sufficient, the agent should prepare materials without unnecessary questions.
  - If data is missing but inferable from uploaded/profile evidence, the agent should suggest a conservative answer and ask for confirmation.
  - If a source is blocked or unsupported, the UI should switch to manual-guided mode rather than failing silently.
  - If the job looks expired/uncertain, show a warning and require the user to choose whether to continue.
  - If the final review contains unsupported claims or empty mandatory fields, submission stays disabled.

### Browser control and interaction design
- Critically, design browser control around the evidence that the current project has no browser automation runtime.
- Therefore the first seamless integration is a transparent browser-interaction state machine, not headless control:
  - Session stores `currentUrl`, `observedPageTitle`, `observedFields`, `stepStatus`, `manualBlockers`, and `lastUserBrowserAction`.
  - UI gives the user explicit “Open page in browser” actions using the listing/application URL from existing search results.
  - User can paste or capture observed form questions/steps back into the session through structured inputs.
  - Agent interprets those observed fields, proposes values from profile/session memory, flags risky answers, and updates a checklist.
- Browser-step states to model:
  - `not_started`, `opened`, `fields_detected`, `answers_prepared`, `awaiting_user_review`, `ready_for_submit`, `submitted`, `blocked`.
- Browser interaction semantics:
  - the agent never claims it clicked or typed anything unless the implementation later gains actual supported tooling,
  - the page should present browser guidance as recommended next actions/checklists,
  - manual observations from the user become new structured inputs for the reasoning services,
  - if future supported browser tooling is introduced, the same persisted browser-state model can back it without rewriting session policies.
- This keeps the first implementation honest, aligned with local capabilities, and still useful as a guided application copilot.

## Implementation Milestones
1. Establish the Auto Apply domain contract.
   - Add shared types/status enums/filter shapes/ground-truth snapshot structures under a new `src/lib/auto-apply/types.ts` and complementary Zod request schemas under `src/lib/auto-apply/schema.ts`.
   - Define explicit session statuses (`idle`, `searching`, `ranking`, `awaiting_selection`, `preparing`, `browser_active`, `awaiting_user_answer`, `ready_for_review`, `submitting`, `submitted`, `blocked`, `failed`, `stopped`) and candidate/application/memory shapes.
   - Include typed AI decision payloads so routes/services return stable JSON for UI rendering.

2. Add persistence models and minimal event logging.
   - Create Mongoose models for `AutoApplySession`, `AutoApplyJobCandidate`, `AutoApplyApplicationDraft`, `AutoApplyMemory`, and `AutoApplyEvent`, using timestamps/indexes consistent with `SearchCampaign`, `JobLead`, and `ApplicationEvent`.
   - Store `sessionGroundTruthSnapshot` as immutable session evidence plus selected IDs for traceability.
   - Add a compact file-reference model if needed (for uploaded resume/additional docs) only if the existing `GeneratedArtifact` model cannot be reused cleanly after inspection by the coding agent.

3. Build the reusable service/orchestration layer.
   - Add `src/lib/auto-apply/` services for session CRUD, search attachment, dedupe/ranking, answer grounding, compliance checks, review package creation, and event logging.
   - Keep route handlers thin, matching the pattern seen in search and profile APIs.
   - Implement a bounded “supervisor” service that coordinates steps and emits structured actions/events, without pretending to run unsupported browser automation.

4. Reuse search discovery and add Auto Apply ranking.
   - Implement session search logic that calls the same validated search pipeline contract used by `/api/search/jobs/stream`, normalizes NDJSON result events into candidate records, records blocked/restricted sources, and preserves source metadata.
   - Add deterministic dedupe/ranking logic based on `dedupeKey`, title/company/location similarity, prompt fit, keyword overlap with selected ground truth, source risk/compliance status, and missing requirements.
   - Support AI Coaching vs Manual Curate gating in the orchestration layer: AI Coaching can auto-shortlist above threshold; Manual Curate must pause until explicit selection.

5. Reuse upload parsing and ground-truth controls.
   - Add resume and optional supporting-document upload routes using `req.formData()` patterns from `/api/ocr`, file validation, parsed text extraction where supported, and metadata persistence.
   - Add session helpers that convert selected profile experiences/projects/skills/facts into a `sessionGroundTruthSnapshot` and enforce “selected-only unless full resume is allowed.”

6. Integrate cover-letter and answer generation.
   - Add an application preparation route that reuses the existing cover-letter route/service by mapping the session’s selected profile items to cover-letter indices.
   - Implement grounded employer-question answering using selected snapshot facts, saved session answers, and conservative classification/clarification rules.
   - Route lightweight extraction/classification work to the mini model preset and complex reasoning/synthesis to the stronger preset after updating `src/lib/ai.ts` defaults/config behavior.

7. Build the `/auto-apply` page and responsive workflow UI.
   - Create a new page using `ModuleShell`, `GlassPanel`, `SectionHeading`, and current gradient/status styles seen in Search/Generate/Control Room.
   - Desktop: left control panel, center jobs/review pane, right activity/preview pane. Mobile: tabbed or accordion sections for Prompt, Jobs, Apply, Activity.
   - Include status pill, prompt/filter form, mode segmented control, file uploads, ground-truth selection, results list with fit reasons/missing requirements, memory panel, clarification modal, review dialog, session history, and final confirmation dialog.

8. Add navigation and final submission gating.
   - Add navbar entry with desktop label `Auto Apply` and mobile short label `Apply`.
   - Ensure submission API requires an explicit confirmation flag and server-side validation that review is complete, source is permitted, resume exists when required, and unresolved uncertainty flags are cleared or user-approved.

9. Add tests and validation.
   - Extend the current unit test harness with deterministic cases for dedupe/ranking, unsupported-answer clarification, session memory reuse boundaries, and submit gating.
   - Run lint and unit tests; repair only within touched surfaces.
