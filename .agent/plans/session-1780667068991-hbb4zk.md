## Summary
Implement a new `/auto-apply` feature as a session-based, compliant, human-in-the-loop application assistant that reuses the existing search stream, cover-letter generation route, profile evidence, MongoDB/Mongoose persistence, and Azure AI wrapper already present in the repository. The smallest coherent path is to add new Auto Apply models, route handlers, orchestration helpers, and a dedicated page that composes the repo’s current UI shell and event-streaming patterns rather than introducing a separate service, external workflow engine, or hidden browser automation stack.

## Intent, Scope, and Non-Goals
The intent is to add an Auto Apply workspace where authenticated users can search for jobs, persist an application session, select explicit ground-truth evidence from their profile, rank jobs, prepare grounded application drafts, answer employer questions safely, review all materials, and explicitly confirm any final submission. This should align with the product rule in the prompt and with existing repository conventions around auth (`src/lib/auth.ts`), Mongoose models (`src/lib/models/*.ts`), streaming search (`src/app/api/search/jobs/stream/route.ts:21-158`, `src/lib/search/server/stream.ts:15-112`), and AI-assisted job reasoning (`src/lib/career/job-understanding.ts:74-112`, `src/lib/career/career-strategist.ts:130-320`).

Scope includes:
- New `src/app/auto-apply/page.tsx` UI with responsive layout, mode selector, status pill, search/filter inputs, upload controls, ground-truth selection, ranked jobs, activity log, review flow, memory panel, and confirmation gating.
- New Auto Apply persistence models for session, job candidate, application draft, and memory, following existing Mongoose patterns from `SearchCampaign`, `JobLead`, and `ApplicationEvent`.
- New `/api/auto-apply/...` routes for session CRUD, uploads, search attach, ranking, draft preparation, clarification, answer saving, review packaging, submission gating, and event retrieval/streaming.
- New orchestration helpers that reuse existing search and cover-letter surfaces and build grounded fit/answer logic on top of current career-analysis utilities.
- Targeted tests added to the existing lightweight Node assertion harness in `tests/career/run-tests.ts`.
- Required implementation comments for major feature work using the repository’s “Motivation vs Logic” convention seen in `src/app/api/search/jobs/stream/route.ts:54-58`, `src/lib/ai.ts:126-133`, and `src/lib/cover-letter.ts:24-29`.

Non-goals:
- No stealth browser automation, captcha solving, login bypassing, account creation, paywall circumvention, or automated submission without explicit user confirmation.
- No replacement of the existing search crawler; reuse the current search API/request/result contract from `src/lib/search/types.ts:35-102` and `src/app/api/search/jobs/stream/route.ts:21-158`.
- No provider migration or new SDK layer; extend the current Azure wrapper in `src/lib/ai.ts` instead of adding another AI client.
- No broad rework of profile, resume builder, or control-room data models beyond what Auto Apply needs to integrate cleanly.
- No assumption of a real browser-runtime dependency already existing in the repo; browser assistance should be modeled behind safe tool interfaces and default to visible/manual-guided preview behavior unless a permitted implementation surface already exists during code execution.

Assumptions grounded in evidence:
- The app uses Next.js App Router pages and route handlers (`src/app/layout.tsx`, `src/app/api/**/route.ts`).
- Auth is cookie/JWT based and route-level (`src/lib/auth.ts:4-47`).
- Data persistence uses Mongoose with simple per-feature schemas (`src/lib/db.ts:17-27`, `src/lib/models/*.ts`).
- Search results already dedupe at stream time via `dedupeKey` plus canonical URLs (`src/lib/search/server/stream.ts:20-89`), but Auto Apply still needs session-local dedupe/ranking records.
- Profile evidence currently lives on the `User.profile` document (`src/lib/models/User.ts:25-46`) and can seed selectable session ground truth.
- The current AI wrapper exposes preset-based text and JSON generation, but model mapping must be updated to satisfy the new `gpt-5.4` / `gpt-5.4-mini` split called for in the prompt (`src/lib/ai.ts:25-33`, `125-133`).

## Implementation Milestones
1. Define Auto Apply domain types and persistence.
   - Add Mongoose models for session, candidate, application draft, and memory with enums matching the requested status vocabulary.
   - Add feature-local TypeScript types and Zod schemas for session creation, filter updates, rank requests, clarification payloads, review payloads, and submission confirmation.
   - Keep IDs/files/memory provenance explicit so the later agents can trace all generated answers back to selected evidence.

2. Extend AI/runtime primitives needed for compliant orchestration.
   - Update `src/lib/ai.ts` preset/model resolution so supervisor-grade reasoning can target `gpt-5.4` while lightweight extraction/classification tasks use `gpt-5.4-mini`, without breaking current callers using `easy`, `hard`, and `document` presets.
   - Add small JSON-safe helper surfaces for Auto Apply classification/ranking tasks using the same pattern as `generateJsonSafe` and existing Zod validation consumers.

3. Build Auto Apply backend orchestration helpers.
   - Create an `auto-apply` lib area containing: session persistence helpers, ground-truth snapshot builder from `User.profile`, session ranking/dedup helpers, question-classification and answer-grounding helpers, compliance evaluator, event logger, and review-packaging helpers.
   - Reuse `streamSearchJobs` for search and `buildFallbackJobUnderstanding`/`extractJobUnderstandingWithLlm` plus `scoreLeadFit`-style logic for fit analysis.
   - Encapsulate “supervisor decisions” as structured JSON objects consumed by routes/UI, with explicit `requiresUserInput`, `allowedActions`, `riskFlags`, and `evidenceRefs` fields.

4. Add Auto Apply API routes.
   - Implement session create/get/update plus subordinate routes for search, rank, upload resume/document, prepare job draft, answer employer question, ask-user, save-answer, review, submit, and events.
   - Follow existing authenticated route conventions (`getAuthFromCookies`, `connectToDatabase`, `NextResponse.json`) and keep long-running/eventful flows on Node runtime routes with force-dynamic where streaming is used.
   - Submission route must require an explicit confirmation flag and fail closed when automation is restricted or required answers are unsupported.

5. Build the `/auto-apply` page and supporting client components.
   - Add a responsive page using the existing shell/panel primitives and Search-page state patterns.
   - Left column: prompt, filters, uploads, ground-truth selection, session controls.
   - Main column: ranked jobs, fit panels, action buttons, review cards.
   - Right/bottom column: activity timeline, browser/app preview placeholder, detected fields, current answer, intervention queue, memory panel, history table.
   - Mobile layout should collapse into tabs/accordion sections while preserving the same session state.

6. Implement clarification, memory, review, and submission gating.
   - Add modal-driven user clarification flow for unsupported/sensitive answers.
   - Persist session memory and optionally reusable memory only when explicitly approved.
   - Generate review bundles that list documents, answers, uncertainty notes, and source metadata before any submission path is enabled.
   - Ensure manual-curate mode blocks prepare/apply progression until the user explicitly selects jobs.

7. Add targeted tests and validate.
   - Add deterministic tests for ranking/dedupe, answer grounding and unsupported-answer escalation, memory save semantics, and submission confirmation gating.
   - Run lint and unit tests; repair only the smallest surfaced issues.
